# DECISIONS

Этот файл хранит важные решения проекта.

Каждое решение должно объяснять не только что выбрано, но и почему.

## 2026-07-03. Telegram как первый интерфейс

Решение:

Начать с Telegram.

Почему:

Telegram уже умеет принимать текст, голос, фото, документы и ссылки. Это самый быстрый путь к MVP, которым основатель сможет пользоваться каждый день.

## 2026-07-03. LifeEvent как главная сущность

Решение:

Строить продукт вокруг `LifeEvent`, а не вокруг отдельных записей.

Почему:

Человек мыслит событиями, проектами, встречами, поездками, темами и контекстами. Это отличает AI Memory OS от AI-диктофона и обычных заметок.

## 2026-07-03. Supabase только как PostgreSQL

Решение:

Использовать Supabase только как managed PostgreSQL через `DATABASE_URL`.

Почему:

Это позволяет быстро получить облачную базу без построения инфраструктуры.

Не используем:

- Supabase Auth;
- Supabase Storage;
- Supabase Edge Functions.

## 2026-07-03. Railway для деплоя

Решение:

Использовать Railway для деплоя первого MVP.

Почему:

Railway подходит для быстрого запуска маленького Node.js-приложения без лишней инфраструктуры.

## 2026-07-03. OpenAI позже

Решение:

Не подключать OpenAI в фундаменте проекта.

Почему:

Сначала нужен стабильный контур: Telegram -> Memory Core -> база. Транскрибация, анализ и умный поиск добавляются после этого.

## 2026-07-03. Не добавлять Redis, workers и Docker

Решение:

На первом этапе не использовать Redis, workers и Docker.

Почему:

Цель — быстрый MVP для одного пользователя, а не промышленная инфраструктура.

## 2026-07-03. Inbox как событие по умолчанию

Решение:

Если система не знает, куда отнести запись, сохранять ее в `Inbox`.

Почему:

Сохранение важнее идеальной классификации. Организацию можно улучшить позже.

## 2026-07-03. Использовать Supabase pooler вместо direct connection

Решение:

Для `DATABASE_URL` использовать Supabase pooler connection string:

```text
postgresql://postgres.project_id:password@aws-0-eu-west-1.pooler.supabase.com:5432/postgres?sslmode=require
```

Почему:

Direct connection вида `db.project_id.supabase.co:5432` был доступен по сети, но Prisma не смог стабильно подключиться к нему на локальной Windows-машине. Pooler connection string успешно прошел `SELECT 1`, после чего Prisma migration была применена.

Что важно:

- Supabase по-прежнему используется только как PostgreSQL;
- Supabase Auth, Storage и Edge Functions не используются;
- пароль базы хранится только в `.env` и не коммитится;
- `sslmode=require` должен оставаться в строке подключения.

## 2026-07-03. MVP Checkpoint 1 подтвержден

Решение:

Зафиксировать первый рабочий MVP checkpoint.

Что подтверждено:

- Telegram bot отвечает на `/start`;
- текстовые сообщения сохраняются в Supabase/Postgres;
- создается или находится пользователь по Telegram ID;
- создается или находится `LifeEvent` с названием `Inbox`;
- текст сохраняется как `MemoryItem` внутри `Inbox`;
- команды `/events` и `/last` работают;
- Telegram command menu и кнопки в чате добавлены.

Почему:

Это первый полностью проверенный цикл продукта:

```text
Telegram -> Memory Core -> Prisma -> Supabase/Postgres -> Read-back в Telegram
```

## 2026-07-03. Добавить безопасное удаление последней записи

Решение:

Добавить команду `/delete_last`, которая удаляет только последнюю `MemoryItem` текущего пользователя.

Почему:

После начала живого тестирования пользователь должен иметь простой способ убрать ошибочную или тестовую запись. Это снижает страх перед capture и не требует сложного управления памятью.

Ограничение MVP:

- удаляется только последняя запись;
- удаление работает только в рамках текущего пользователя;
- массовое удаление, корзина и восстановление не добавляются на этом этапе.

## 2026-07-03. Railway deploy через long polling

Решение:

Запускать Telegram MVP на Railway через long polling, без webhook и без отдельного HTTP server.

Почему:

Для первого рабочего MVP это самый простой и быстрый способ получить постоянно работающего бота. Нам не нужно поднимать web app, настраивать публичный endpoint, SSL, webhook routing или отдельную инфраструктуру.

Что важно:

- в Railway должны быть переменные `BOT_TOKEN` и `DATABASE_URL`;
- `DATABASE_URL` ведет в Supabase/Postgres через pooler connection string;
- секреты не коммитятся в Git;
- при long polling должен быть активен только один экземпляр бота;
- если локальный бот и Railway bot работают одновременно, Telegram может вернуть `409 Conflict`;
- при такой ошибке нужно остановить лишний экземпляр и оставить один активный процесс.

Ограничение MVP:

Long polling подходит для текущего этапа и одного основателя-пользователя. Webhook можно рассмотреть позже, когда появится реальная причина: web-интерфейс, несколько сервисов, более строгий production-контур или отдельный HTTP API.
## 2026-07-04. LLM как сервис понимания входящих сообщений

Решение:

Подключать LLM отдельным слоем `src/ai`, а не внутри Telegram handlers.

Первый LLM use case:

Классифицировать входящее сообщение и превращать его в структурированный `OpenCycle` с полями `type`, `title`, `context`, `area`, `urgency`, `importance`, `energy`, `estimatedMinutes`, `dueDate` и `reason`.

Почему:

AI Memory OS не должен превращаться в обычный AI chat. LLM нужен как сервис понимания входящих сообщений: он помогает системе увидеть задачу, мысль, покупку, идею, обещание или заметку внутри сырого текста.

Архитектурная граница:

- Telegram принимает сообщение;
- Memory Core сохраняет данные;
- AI layer классифицирует текст;
- Supabase/Postgres хранит результат;
- Telegram показывает пользователю простой ответ.

Что важно:

- Supabase остается основной базой данных;
- OpenAI является текущим provider, но не должен протекать в бизнес-логику;
- provider подключается через adapter pattern;
- `transcribeVoice` пока остается TODO;
- `buildMorningFocus` готовится как будущий use case, но не включается в текущий поток.
## 2026-07-04. Input Normalization вместо отдельной Transcription-системы

Решение:

Не создавать отдельный модуль `Transcription` как центральную часть системы.

Вместо этого вводится единый слой `Input Normalization`.

Почему:

Голос — только один из будущих источников. Если сделать транскрибацию центральной архитектурной сущностью, проект начнет мыслить аудио-файлами, а не универсальным входом в память.

Правильная модель:

```text
Input
  -> Input Normalization
  -> Normalized Text
  -> LLM Classification
  -> Context
  -> OpenCycle / MemoryItem
```

Что важно:

- текст Telegram уже является нормализуемым входом;
- voice позже нормализуется через транскрибацию;
- image позже нормализуется через OCR или Vision;
- document/PDF позже нормализуется через text extraction;
- link позже нормализуется через title, main text и summary;
- Memory Core работает с `NormalizedInput`, а не с Telegram, PDF, voice или image напрямую.

Результат:

Новые источники вроде pendant, email, calendar, WhatsApp или browser добавляются в `src/input`, не ломая остальную архитектуру.
## 2026-07-04. Text -> OpenCycle только после сохранения MemoryItem

Решение:

Подключать LLM-классификацию текстовых сообщений только после успешного сохранения `MemoryItem`.

Поток:

```text
Text
  -> normalizeText
  -> save MemoryItem
  -> classifyInput, если включено
  -> save OpenCycle
```

Почему:

Пользователь не должен терять запись из-за ошибки OpenAI, сетевого сбоя или неправильного JSON. Capture важнее классификации.

Ограничение:

LLM-классификация управляется env-флагом `LLM_CLASSIFICATION_ENABLED`. Если флаг выключен или `OPENAI_API_KEY` не задан, бот сохраняет только `MemoryItem` и работает как раньше.
## 2026-07-04. Prompt tuning для OpenCycle type

Решение:

Prompt для `classifyInput` должен быть на русском языке, потому что первый пользовательский поток проекта сейчас русскоязычный.

Почему:

Если входные сообщения на русском, англоязычный prompt с русскими вставками хуже передает продуктовые нюансы: покупки, бытовые дела, обещания, мысли и заметки. Кроме того, русские примеры должны храниться в нормальном UTF-8, без mojibake.

Что уточнено:

- `PURCHASE` имеет приоритет для сообщений с намерением купить, заказать, докупить, взять или приобрести;
- `TASK` используется для конкретных действий, которые не являются покупками;
- `PROMISE` используется для обещаний и обязательств перед другими людьми;
- `IDEA` используется для продуктовых, проектных и творческих идей;
- `THOUGHT` используется для размышлений и наблюдений;
- `NOTE` используется для фактов, кодов, дат и справочной информации;
- `OTHER` используется только когда ничего не подходит.

Добавлены русские few-shot примеры для всех текущих типов: `PURCHASE`, `TASK`, `PROMISE`, `IDEA`, `THOUGHT`, `NOTE`, `OTHER`. Также покрыты ожидания ответа и открытые вопросы: пока они мапятся в существующие типы `NOTE` и `THOUGHT`.

Не добавляем пока новые enum-типы вроде `WAITING` и `QUESTION`, чтобы не усложнять схему данных до реальной необходимости.
## 2026-07-04. Показывать OpenCycles через Telegram

Ограничение:

Это только просмотр. Редактирование, закрытие, перенос и фильтрация OpenCycles будут добавлены позже.
## 2026-07-04. Закрывать OpenCycle через closedAt

Решение:

Не удалять завершенные `OpenCycle`, а закрывать их через поле `closedAt`.

Почему:

Закрытие цикла является частью памяти. Если удалить запись, система потеряет историю того, что было сделано или больше не требует внимания.

Ограничение MVP:

Команда `/close_cycle` закрывает последний открытый цикл. Выбор конкретного цикла по номеру будет добавлен позже, если понадобится.

## 2026-07-04. Reclassify last MemoryItem

Решение:

Добавить `/reclassify_last`, чтобы последнюю исходную запись можно было заново прогнать через текущий prompt.

Почему:

Prompt будет улучшаться. Старые `OpenCycle` могут быть созданы старой версией prompt, поэтому нужен простой способ исправить последнюю запись без ручного удаления в базе.

## 2026-07-04. Natural Intent Layer перед созданием нового OpenCycle

Решение:

Добавить слой `Natural Intent`, который определяет, является ли входящее сообщение новой записью или действием над уже существующей памятью.

Почему:

Реальный пользователь не должен помнить технические команды. Он будет говорить естественно: `помидоры уже купил`, `письмо в ЖЭК написал`, `удали эту запись`, `что я говорил про Японию`. Если система на каждую такую фразу будет создавать новый `OpenCycle`, продукт быстро станет неудобным.

Новый поток:

```text
Normalized Text
  -> save MemoryItem
  -> classifyIntent
  -> выполнить намерение или безопасно сохранить как память
```

Что реализовано сейчас:

- `CREATE_OPEN_CYCLE` создаёт новый `OpenCycle` через существующий `classifyInput`;
- `CREATE_MEMORY` сохраняет только `MemoryItem`, без нового открытого цикла;
- `CLOSE_OPEN_CYCLE` ищет похожий открытый цикл среди последних открытых циклов пользователя и закрывает его через `closedAt`;
- `DELETE_MEMORY`, `UPDATE_MEMORY`, `SEARCH_MEMORY`, `MOVE_TO_CONTEXT` пока только распознаются, но не выполняются автоматически.

Почему не делаем автоматическое удаление сразу:

Удаление и изменение старых записей опаснее закрытия цикла. Если LLM ошибётся, пользователь может потерять важную память. Для таких действий позже нужен безопасный confirmation flow: бот нашёл запись, показал её и спросил подтверждение.
