# ARCHITECTURE

## Принцип MVP

Архитектура первого этапа должна быть простой.

Мы не строим систему на миллион пользователей. Мы строим продукт, которым один человек сможет пользоваться каждый день.

Минимальная схема:

```text
Telegram bot
  -> Memory Core
  -> Prisma
  -> Supabase/Postgres
  -> Search/Recall
```

## Memory Flow

Жизненный цикл данных важнее списка слоев.

Пример для голосового сообщения:

```text
Пользователь
  -> отправил голос в Telegram
  -> система приняла входящее сообщение
  -> сохранила оригинальный Asset
  -> создала MemoryItem
  -> определила или создала LifeEvent
  -> при неуверенности положила запись в Inbox
  -> позже обновила summary события
  -> запись стала доступна через Search/Recall
```

Пример для текстовой мысли:

```text
Пользователь
  -> отправил текст
  -> Memory Core создал MemoryItem
  -> MemoryItem попал в LifeEvent или Inbox
  -> пользователь получил подтверждение
  -> запись можно найти позже
```

Главное правило потока:

Сначала сохранить. Потом понять. Потом связать.

## Что используем

На первом этапе:

- Telegram — первый интерфейс;
- Railway — деплой приложения;
- Supabase — PostgreSQL-база;
- Prisma — ORM и схема данных;
- OpenAI — позже, не в фундаменте.

Supabase используется только как база данных.

Не используем:

- Supabase Auth;
- Supabase Storage;
- Supabase Edge Functions.

## Что не добавляем на первом этапе

Не добавлять без отдельного решения:

- Redis;
- workers;
- Docker;
- web app;
- mobile app;
- сложную event-driven архитектуру;
- отдельные микросервисы.

## Слои системы

### Interface Layer

Принимает пользовательский ввод.

Первый интерфейс:

- Telegram.

Будущие интерфейсы:

- Web;
- Mobile;
- Voice Pendant;
- Smart Watch;
- API.

Интерфейсы не должны содержать главную бизнес-логику памяти.

### Memory Core

Главное ядро продукта.

Отвечает за:

- создание пользователя;
- создание и выбор `LifeEvent`;
- создание `MemoryItem`;
- связь с `Asset`;
- будущий поиск;
- будущий recall;
- правила работы с `Inbox`.

### Data Layer

Работает с PostgreSQL через Prisma.

Физически база находится в Supabase.

Подключение идет через `DATABASE_URL`.

## Основные сущности

### User

Пользователь системы.

На первом этапе пользователь может быть связан с Telegram ID.

### LifeEvent

Главная сущность продукта.

Это событие жизни, проект, встреча, поездка, тема или рабочий контекст.

### MemoryItem

Отдельная запись внутри `LifeEvent`.

Примеры:

- текст;
- голосовая заметка;
- фото;
- документ;
- ссылка;
- идея;
- задача.

### Asset

Оригинальный файл или объект.

Примеры:

- аудио;
- фото;
- документ;
- Telegram file id;
- URL.

## Inbox

`Inbox` — событие по умолчанию.

Если система не понимает, куда отнести запись, она сохраняет ее в `Inbox`.

Это важно: память сначала должна сохранять, а уже потом помогать организовывать.

## Как масштабироваться позже

Масштабирование должно происходить через ясные границы, а не через преждевременную инфраструктуру.

Позже можно будет вынести:

- транскрибацию в worker;
- обработку документов в отдельный процесс;
- хранение файлов в object storage;
- поиск в отдельный search service;
- новые интерфейсы поверх Memory Core.

Но на первом этапе все остается простым.
