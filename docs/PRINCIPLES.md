# PRINCIPLES

Этот документ — конституция AI Memory OS.

Если архитектурное или продуктовое решение конфликтует с этими принципами, решение нужно пересмотреть.

## Принцип 1. Capture first. Organize later.

Пользователь никогда не должен думать, куда сохранить информацию.

Сначала сохранить.

Потом понять.

Потом связать.

Организация памяти никогда не должна мешать фиксации мысли.

## Принцип 2. Никогда не потерять запись

Если система не уверена в контексте, она сохраняет запись в `Inbox`.

Лучше сохранить неидеально, чем потерять важную мысль.

Удаление, перезапись и автоматическая классификация должны быть осторожными.

## Принцип 3. Контекст важнее файла

Голос, фото, PDF или ссылка сами по себе имеют ограниченную ценность.

Они становятся памятью, когда система понимает, к какому событию, проекту, человеку, встрече или решению они относятся.

AI Memory OS сохраняет не просто данные, а контекст жизни.

## Принцип 4. Жизнь важнее файлов

Пользователь мыслит не папками и расширениями файлов.

Пользователь мыслит событиями, проектами, людьми, поездками, встречами и темами.

Поэтому продукт должен строиться вокруг жизненного контекста, а не вокруг формата входных данных.

## Принцип 5. LifeEvent важнее MemoryItem

`MemoryItem` — это фрагмент.

`LifeEvent` — это смысловой контекст.

Поиск, recall и будущий AI-анализ должны усиливать связи между записями и событиями.

## Принцип 6. AI помогает, но не забирает контроль

AI может предлагать событие, теги, связи, summary и ответы.

Но пользователь должен иметь возможность исправить, переместить, переименовать или удалить результат.

Автоматизация не должна превращаться в черный ящик.

## Принцип 7. Любое усложнение должно быть оправдано

Redis, workers, Docker, отдельные сервисы, web app, mobile app и сложные пайплайны добавляются только тогда, когда простая архитектура перестала справляться.

Красота архитектуры не важнее работающего MVP.

## Принцип 8. Пользователь должен помнить меньше

Система должна уменьшать количество действий, которые пользователь обязан выполнить.

Если продукт требует слишком много ручной организации, он перестает быть внешней памятью.

## Принцип 9. Telegram — интерфейс, не продукт

Telegram нужен для быстрого MVP и ежедневного capture.

Настоящий продукт — Memory Core и контекстная память под ним.

Будущие интерфейсы должны подключаться к тому же ядру.

## Принцип 10. Простота — актив проекта

На раннем этапе простая система лучше сложной.

Каждый новый слой должен отвечать на реальную боль, а не на воображаемый масштаб.
## Принцип 8. Normalize input first

Любой источник информации сначала нормализуется, а потом вся система работает только с единым форматом данных.

Telegram, голос, фото, PDF, ссылка, email, медальон и будущие интеграции не должны расползаться по бизнес-логике.

Сначала:

```text
Input -> Input Normalization -> NormalizedInput
```

И только потом:

```text
NormalizedInput -> Memory Core -> LLM Classification -> OpenCycle / MemoryItem
```

Это защищает проект от хаоса интеграций и позволяет добавлять новые источники без переписывания ядра памяти.
