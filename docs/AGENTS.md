# AGENTS

Этот файл обязателен для любых AI-агентов, которые работают с проектом AI Memory OS.

Документ не привязан к конкретной модели. Через время инструменты могут измениться, но правила работы должны остаться понятными.

## Роль AI-агента

Любой AI-агент в этом проекте должен помогать развивать AI Memory OS маленькими, проверяемыми шагами.

Агент может выполнять разные роли:

- архитектор;
- разработчик;
- ревьюер;
- технический аудитор;
- помощник по документации;
- исследователь альтернативных решений.

Конкретная роль зависит от задачи пользователя.

## Перед началом работы

Агент обязан прочитать:

- `README.md`;
- `docs/PROJECT.md`;
- `docs/PRINCIPLES.md`;
- `docs/VISION.md`;
- `docs/ARCHITECTURE.md`;
- `docs/DECISIONS.md`;
- `docs/GLOSSARY.md`.

Если задача касается продукта или архитектуры, сначала нужно свериться с этими документами.

## Главные правила

- Не усложнять.
- Не добавлять инфраструктуру заранее.
- Делать минимальные изменения.
- Объяснять свои решения.
- Проверять результат, если была реализация.
- Не предлагать рефакторинг ради рефакторинга.
- Не переписывать весь проект без необходимости.
- Сохранять принцип: сначала capture, потом organization.
- Помнить, что AI Memory OS сохраняет контекст жизни, а не просто файлы.

## Архитектурные правила

- Telegram — только первый интерфейс.
- Главная сущность продукта — `LifeEvent`.
- `MemoryItem` всегда находится внутри жизненного контекста.
- `Asset` хранит оригинальный файл или объект.
- `Inbox` используется как событие по умолчанию.
- Все интерфейсы должны работать через Memory Core.
- Telegram не должен владеть логикой памяти.
- Memory Core должен быть независим от Telegram.
- База данных подключается через Prisma.
- Supabase используется только как PostgreSQL.
- Railway используется для деплоя.
- OpenAI подключается позже.

## Если агент делает аудит

Аудит должен быть критическим, но практичным.

Формат ответа:

1. Что хорошо.
2. Что вызывает риск.
3. Что стоит упростить.
4. Что обязательно исправить сейчас.
5. Что можно оставить на потом.
6. Какую следующую задачу дать исполнителю.

Не нужно соглашаться автоматически. Если решение слабое, нужно сказать прямо и предложить вариант лучше.

## Если агент пишет код

Код можно писать только если задача явно требует реализации.

Перед изменением кода нужно понять:

- какая проблема решается;
- какой минимальный набор файлов нужно изменить;
- как проверить результат;
- не нарушает ли изменение принципы MVP.

## Правила коммуникации

Агент должен ясно сообщать:

- что сделано;
- какие файлы изменены;
- что проверено;
- что не было сделано;
- какие есть риски.

Нельзя заявлять, что что-то работает, если это не проверено.
