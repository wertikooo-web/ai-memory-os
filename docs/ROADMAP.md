## Этап 4. Railway Deploy

Статус: выполнено.

MVP задеплоен на Railway и подтвержден в живом Telegram-боте.

Текущая схема:

```text
Railway -> Node.js/grammY long polling -> Supabase/Postgres
```

Цель этапа достигнута: бот работает постоянно, без запуска с локального компьютера.

Важное ограничение: при long polling должен работать только один экземпляр бота. Локальный запуск и Railway одновременно могут вызвать `409 Conflict` от Telegram.

## MVP Checkpoint 3. Text -> OpenCycle

Статус: реализовано в коде, включается через env-флаг.

Что сделано:

- текст проходит через `Input Normalization`;
- `MemoryItem` сохраняется первым;
- LLM-классификация запускается только если включен `LLM_CLASSIFICATION_ENABLED`;
- результат классификации сохраняется как `OpenCycle`;
- если LLM падает, запись не теряется.
## MVP Checkpoint 4. OpenCycle visibility

Статус: реализовано.

Что сделано:

- добавлена команда `/open_cycles`;
- бот показывает последние `OpenCycle` текущего пользователя;
- можно проверить, что LLM-классификация действительно сохраняется в Supabase/Postgres;
- Telegram command menu и reply keyboard обновлены.

Почему это важно:

До этого пользователь видел только короткий ответ `Понял как: ...`, но не видел сохраненный structured результат. Теперь можно проверять память прямо из Telegram.
## MVP Checkpoint 5. Reclassify and close cycles

Статус: реализовано.

Что сделано:

- `/reclassify_last` заново классифицирует последнюю запись через текущий русский prompt;
- если `OpenCycle` для записи уже есть, он обновляется;
- `/close_cycle` закрывает последний открытый цикл;
- закрытый цикл не удаляется, а получает `closedAt`;
- `/open_cycles` показывает только незакрытые циклы.
